Лабораторно-практическая работа №5

Структурные паттерны проектирования: Компоновщик, Адаптер и Фасад

Выполнил: Митанов Д.В  Группа: 2407СА1  Проверил: Макаров М. С.


Цель работы

Изучить структурные паттерны проектирования GoF на примере создания приложения, моделирующего работу файлового менеджера с поддержкой разных файловых систем и облачных хранилищ. В рамках работы необходимо освоить применение паттерна Компоновщик (Composite) для построения иерархии файлов и папок, паттерна Адаптер (Adapter) для организации единого интерфейса доступа к различным типам файловых систем и паттерна Фасад (Facade) для предоставления удобного интерфейса выполнения типовых операций, например синхронизации данных и резервного копирования.


Задание


Требовалось разработать консольное приложение, которое имитирует работу файлового менеджера. Программа должна обеспечивать:

создание иерархической структуры файловой системы, включающей файлы и папки;

использование паттерна Composite, при котором файлы выступают как конечные элементы структуры, а папки могут содержать в себе другие объекты файловой системы;

выполнение рекурсивных операций обхода структуры, включая подсчёт суммарного размера папок, удаление элементов и копирование файлов;

подключение различных файловых систем через единый интерфейс с применением паттерна Adapter;

предоставление простого способа выполнения типовых операций через паттерн Facade, например для синхронизации локальной директории с облачным хранилищем и создания резервных копий.



Теоретическое обоснование

Структурные паттерны проектирования применяются для упрощения связей между объектами и классами системы. Они помогают сделать архитектуру более гибкой, понятной и удобной для дальнейшего расширения.

Паттерн Компоновщик (Composite) предназначен для представления объектов в виде древовидной структуры. Благодаря этому клиентский код может одинаково работать как с отдельными объектами, так и с их группами. В файловой системе это особенно удобно, так как и файлы, и папки можно представить как элементы одной общей структуры. При этом папки могут содержать другие элементы, а файлы являются конечными узлами дерева.

Паттерн Адаптер (Adapter) используется тогда, когда необходимо обеспечить совместную работу классов с несовместимыми интерфейсами. Он преобразует один интерфейс в другой, который ожидается клиентским кодом. В данной лабораторной работе этот паттерн нужен для того, чтобы разные файловые системы можно было использовать через единый набор методов.

Паттерн Фасад (Facade) предоставляет простой интерфейс для работы со сложной подсистемой. Он скрывает детали взаимодействия между несколькими классами и позволяет выполнять типовые операции через небольшое количество понятных методов. Благодаря этому работа с системой становится проще, а код клиента — чище и понятнее.

Использование перечисленных паттернов позволяет снизить связанность между компонентами, улучшить организацию программы и упростить дальнейшее добавление нового функционала


Описание выполненной работы


В ходе выполнения лабораторной работы было разработано консольное приложение, моделирующее работу файлового менеджера.

На первом этапе был реализован паттерн Composite. Для этого создан общий абстрактный класс, представляющий элемент файловой системы. От него были унаследованы два конкретных класса: один для файлов, другой для папок. Папка содержит коллекцию вложенных элементов и поддерживает операции добавления и удаления дочерних объектов. Также был реализован метод, позволяющий вычислять общий размер папки путём рекурсивного обхода всех входящих в неё файлов и подпапок.

На втором этапе был реализован паттерн Adapter. Для этого был определён единый интерфейс работы с файловой системой, включающий такие операции, как просмотр содержимого каталога, чтение файла, запись данных и удаление элементов. Затем был создан адаптер, который позволяет использовать ранее разработанную модель файловой системы через этот общий интерфейс.

На третьем этапе был реализован паттерн Facade. Для этого создан специальный класс, предоставляющий упрощённый набор методов для выполнения типовых операций. С его помощью были организованы функции синхронизации папок и создания резервных копий. Этот класс скрывает внутреннюю сложность рекурсивной обработки файловой структуры и работы с адаптерами различных файловых систем.

В основной части программы была продемонстрирована работа всех разработанных компонентов. Была сформирована иерархия файлов и папок, выполнен подсчёт общего размера корневой директории, а также показана работа механизмов синхронизации и резервного копирования между различными файловыми системами.



Результат выполненной работы


В результате выполнения лабораторной работы было создано консольное приложение, моделирующее работу файлового менеджера.

Проверка программы показала, что использование паттерна Composite позволяет удобно представлять древовидную структуру файловой системы и одинаково обрабатывать как отдельные файлы, так и целые папки. Применение паттерна Adapter обеспечивает единый интерфейс для взаимодействия с разными типами файловых систем, благодаря чему основные операции можно выполнять одинаковым способом независимо от особенностей конкретного хранилища. Паттерн Facade упрощает использование системы, предоставляя клиентскому коду простой и понятный интерфейс для выполнения сложных действий, таких как синхронизация данных и создание резервных копий.

Таким образом, поставленная цель работы была достигнута, а применение структурных паттернов проектирования подтвердило их полезность при разработке гибких и расширяемых приложений.
